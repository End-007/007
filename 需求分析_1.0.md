### 学管平台需求分析报告（初步）

**项目名称**：学管平台  
**覆盖范围**：仅限本学院（不涉及跨学院或全校数据），聚焦学院内部教学、科研、学生管理等核心业务。  
**核心目标**：构建一个统一、可扩展的管理平台，实现数据的分层收集、统计分析与可视化展示，支持多角色权限管理，避免信息孤岛，提升学院运营效率。

---

### 一、业务背景与目标

当前学院数据分散在教务、科研、学工等部门，缺乏统一入口，导致统计效率低、决策依据不足。本平台旨在：

1. **整合数据**：覆盖教学（课程、成绩）、科研（项目、成果）、学习（进度、行为）等全维度数据。
2. **分层管理**：按学院 → 系 → 班级 → 个人四级数据层级，支持精细化统计。
3. **可视化决策**：通过数据大屏实时展示整体运营态势，辅助管理层快速洞察。
4. **角色化服务**：针对教师、学生、部门负责人等角色定制功能，确保权限安全。

> _注：平台设计严格限定在本学院范围，不预留跨学院扩展接口。_

---

### 二、用户角色与权限定义

平台采用 **RBAC（基于角色的访问控制）** 模型，不同角色权限如下表：

| 角色                                 | 权限范围                                      | 典型功能场景                                                                     |
| ------------------------------------ | --------------------------------------------- | -------------------------------------------------------------------------------- |
| **学院管理员**（院长/教务主管）      | 全平台数据访问、角色管理、数据大屏全局控制    | 查看全院级数据大屏；审批部门数据权限；生成月度/季度综合报告；管理教师/学生账号。 |
| **部门负责人**（教务科长、科研处长） | 本部门数据管理（教务/科研），可下钻至系级数据 | 教务：录入课程表、统计不及格率；科研：申报项目、跟踪成果；导出本部门报表。       |
| **教师**                             | 个人教学数据录入、学生进度查看、成绩管理      | 录入课程大纲、布置作业、批改作业；查看所授班级学生学习进度；导出学生成绩报表。   |
| **学生**                             | 个人学习数据查看、基本信息维护、进度反馈      | 查看个人课程进度、成绩、科研参与情况；提交学习反馈；维护联系方式等基本信息。     |
| **系统维护员**（IT 支持）            | 系统配置、数据备份、权限审计                  | 管理用户角色；监控平台性能；处理数据异常。                                       |

> _权限设计原则：_
>
> - 数据层级隔离（如教师仅能查看本人所授班级数据，无法访问其他系数据）。
> - 敏感数据（如成绩）默认仅限角色自身或上级角色访问。
> - 无“超级管理员”角色，避免权限滥用。

### 三、核心功能需求

#### 1. **数据分层录入与统计**

- **数据层级**（学院 → 系 → 班级 → 个人）：
  - **学院级**：全院总览数据（如总学生数、科研项目总数）。
  - **系级**：按院系划分（如计算机系课程完成率、科研产出量）。
  - **班级级**：按班级统计（如某班级平均成绩、学习进度达标率）。
  - **个人级**：学生/教师专属数据（如学生个人学习轨迹、教师科研成果）。
- **数据类型**：
  | 类别 | 数据内容示例 | 录入方 |
  |--------|-----------------------------------------------------------------------------|----------------|
  | 教学 | 课程表、作业提交率、考试成绩、教学评价 | 教师、教务部门 |
  | 科研 | 科研项目申报、论文发表、专利成果、经费使用 | 科研部门、教师 |
  | 学习 | 在线学习时长、作业完成进度、课堂考勤、学习行为分析（如平台活跃度） | 系统自动采集+教师录入 |
  | 基础信息 | 学生档案（学号、专业、联系方式）、教师信息（职称、研究方向）、班级信息 | 管理员、部门负责人 |

- **统计能力**：
  - 自动生成多维度报表（如“2026 年 3 月教学进度统计：计算机系课程完成率 92%”）。
  - 支持时间筛选（日/周/月/季/年），显示数据变化趋势（如“近 3 月科研项目增长 15%”）。
  - 部门间协作：教务部门录入课程数据后，科研部门可关联“教学-科研”交叉分析（如“参与科研项目的学生课程通过率提升 20%”）。

#### 2. **数据大屏（核心展示层）**

- **定位**：学院管理层实时决策中枢，嵌入平台首页。
- **关键指标**（动态更新，支持下钻）：
  | 指标类别 | 具体指标示例 | 展示形式 |
  |------------|-----------------------------------------------------------------------------|------------------|
  | **教学** | 课程完成率、不及格率、平均成绩、教师教学评价分 | 折线图+热力地图 |
  | **科研** | 项目申报数、论文发表量、经费到账率、成果转化率 | 饼图+柱状图 |
  | **学习** | 学生学习进度达标率、在线学习时长、作业提交率、异常行为预警（如长期未登录） | 仪表盘+趋势线 |
  | **综合** | 全院学生总数、班级分布、教学-科研关联度（如“85%学生参与至少 1 项科研”） | 全局概览卡片 |
- **交互要求**：
  - 点击任一指标可下钻至下一层级（如点击“课程完成率”→ 跳转至系级明细）。
  - 支持导出为 PDF/Excel，用于会议汇报。

#### 3. **角色化功能实现**

- **教师层面**：
  - 个人工作台：显示所授课程进度、学生学习状态（如“张三本周作业未提交”）。
  - 便捷录入：批量导入成绩、一键生成学生成绩单。
- **学生层面**：
  - 个人中心：可视化学习进度（如“本学期完成 85%课程”）、成绩查询、科研机会推送（如“有空缺科研助理岗位”）。
  - 互动反馈：提交学习困难申请，触发教师跟进。
- **部门协同**（教务 vs 科研）：
  - 教务部门录入课程数据后，系统自动标记“高潜力学生”（如成绩优异且活跃），科研部门可定向邀请参与项目。

### 四、数据模型设计（简化版）

- **核心实体**：
  - `学院`（ID, 名称）→ `系`（ID, 名称, 所属学院）→ `班级`（ID, 名称, 所属系）→ `学生`（ID, 姓名, 班级 ID）。
  - `课程`（ID, 名称, 系 ID, 教师 ID）→ `学习进度`（学生 ID, 课程 ID, 完成度, 时间戳）。
  - `科研项目`（ID, 项目名, 负责人 ID, 状态）→ `成果`（项目 ID, 论文/专利, 发表时间）。
- **数据流向**：  
  `部门录入 → 平台统一存储 → 分层统计 → 大屏展示`  
  _示例：教师录入成绩 → 系级统计“计算机系平均成绩” → 学院级大屏显示“全院平均成绩 90.5”_

### 五、非功能需求

1. **安全性**：
   - 数据加密传输（HTTPS），敏感字段（如成绩）存储脱敏。
   - 操作日志审计（记录关键数据修改）。
2. **性能**：
   - 数据大屏响应时间 ≤ 2 秒（支持 500+并发用户）。
   - 统计报表生成时间 ≤ 5 分钟（大数据量下）。
3. **扩展性**：
   - 仅预留基础数据结构（如“其他业务”字段），不支持新增业务模块（确保聚焦本学院）。
4. **易用性**：
   - 界面符合学院用户习惯（如教师端简化操作，学生端移动端适配）。

### 总结

本需求分析聚焦“**本学院数据整合**”，通过**分层统计+角色化权限+数据大屏**，解决部门数据割裂问题。平台不追求复杂功能，而是确保核心场景（教学、科研、学习）的流畅闭环，为学院管理层提供“一屏知全局”的决策支持。下一步将细化数据字典与界面原型，进入设计阶段。

> _注：本分析基于用户描述初步提炼，未包含技术栈选型（如后端语言、数据库），后续需与开发团队对齐。_
